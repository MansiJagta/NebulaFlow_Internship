# Nebula-Flow Complete Schema Blueprint (Updated)

This document contains the updated comprehensive MongoDB schema design for the Nebula-Flow platform. Based on your latest requirements, we are pivoting *away* from the Gantt Chart, and replacing it with a robust **Calendar View** that will display Task Deadlines, Milestones, and new Team Meetings.

## 1. Project Hierarchy & Roles

### `Workspace` (Project) Model
**What it is:** The container for a single project.
**Why we use it:** Fixes the "different PM per project" requirement by defining roles inside a `members` array. Ensures you can store collaborators of a specific project in the DB, allowing task assignment and direct messaging to be correctly filtered.
```javascript
const workspaceSchema = new mongoose.Schema({
  name: { type: String, required: true },
  description: { type: String, default: '' },
  members: [{
    userId: { type: mongoose.Schema.Types.ObjectId, ref: 'User' },
    role: { type: String, enum: ['pm', 'collaborator'], default: 'collaborator' }
  }],
  githubConfig: { repoOwner: { type: String }, repoName: { type: String } }
}, { timestamps: true });
```

## 2. Roadmaps replacing Gantt with Calendar

### `Meeting` Model [NEW]
**What it is:** Scheduled events generated by the team (eventually from the Slack chat module).
**Why we use it:** You requested the ability to schedule meetings in chat that automatically sync to all collaborators' calendars. This model stores those events so the Calendar View can fetch them alongside Tasks.
```javascript
const meetingSchema = new mongoose.Schema({
  workspaceId: { type: mongoose.Schema.Types.ObjectId, ref: 'Workspace', required: true },
  title: { type: String, required: true }, // e.g. "Frontend Sync"
  description: { type: String, default: '' },
  startTime: { type: Date, required: true },
  endTime: { type: Date, required: true },
  organizerId: { type: mongoose.Schema.Types.ObjectId, ref: 'User' },
  attendees: [{ type: mongoose.Schema.Types.ObjectId, ref: 'User' }], // Array of collaborators
  location: { type: String, default: '' }, // Virtual or physical location
  slackChannelId: { type: mongoose.Schema.Types.ObjectId, ref: 'Channel' } // Link back to the chat where it was scheduled
}, { timestamps: true });
```

### `Milestone` Model
**What it is:** A macro-level phase on the project (e.g., "Deployment Week").
**Why we use it:** Still useful for the Calendar! They will render as multi-day banner events spanning across the top of the calendar.
```javascript
const milestoneSchema = new mongoose.Schema({
  workspaceId: { type: mongoose.Schema.Types.ObjectId, ref: 'Workspace', required: true },
  name: { type: String, required: true },
  description: { type: String, default: '' },
  startDate: { type: Date, required: true },
  endDate: { type: Date, required: true }
}, { timestamps: true });
```

## 3. Agile Jira Tracking & Role-based Filters

### Role-Based Jira Filters (Frontend Logic)
As requested, we will update the Jira UI to include a **"My Tasks" / "Team Tasks"** toggle:
* **Collaborator View:** Defaults to "My Tasks". They can switch to "Team Tasks" to see what everyone is doing.
* **PM View:** Defaults to "Team Tasks". They will get advanced dropdowns to filter by Assignee, Priority, and Status.

### `Issue` Model (Granular Tasks)
**Why we use it:** Tracks developer day-to-day work. The `dueDate` field is critical because it will plot the task deadline as an event block on the new **Calendar View**.
```javascript
const issueSchema = new mongoose.Schema({
  workspaceId: { type: mongoose.Schema.Types.ObjectId, ref: 'Workspace', required: true },
  sprintId: { type: mongoose.Schema.Types.ObjectId, ref: 'Sprint' },
  issueKey: { type: String, required: true },
  title: { type: String, required: true },
  description: { type: String, default: '' },
  status: { type: String, enum: ['backlog', 'todo', 'in-progress', 'review', 'done'] },
  priority: { type: Number, enum: [1, 2, 3] },
  type: { type: String, enum: ['story', 'task', 'bug'], default: 'task' },
  storyPoints: { type: Number, default: 0 },
  dueDate: { type: Date }, // Will render on the new Calendar
  assigneeUserId: { type: mongoose.Schema.Types.ObjectId, ref: 'User' },
  reporterUserId: { type: mongoose.Schema.Types.ObjectId, ref: 'User' },
  githubPrUrl: { type: String }, // Links to external GitHub PRs
}, { timestamps: true });
```

## 4. GitHub Data: Storing vs Real-Time Fetching

### What we Store in the Database:
1. **Workspace GitHub Mapping:** We store exactly *which* repository belongs to a Workspace (`Workspace.githubConfig.repoOwner` and `repoName`). 
2. **Issue to PR Links:** When a PR is created for an issue, we store the `githubPrUrl` on the MongoDB `Issue` model so the Jira card can have a clickable GitHub button.

### What we Fetch Real-Time directly from GitHub API:
1. **GitHub Page Data:** When you load the "GitHub" or "Performance" tab, the frontend asks the backend to make a *live HTTP request* to `api.github.com/repos/{owner}/{repo}/commits` using your OAuth token. 
2. **Live Stats:** We will dynamically crunch number of open issues, recent PRs, and commit histories on the fly. 
*(Note: As requested, we will create mock/dummy responses for these live endpoints to simulate real behavior without needing a fully authenticated real repository right now, keeping it prototype-ready!)*
